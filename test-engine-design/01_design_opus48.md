# Opus 4.8 设计：ACEE —— Agentic Capability Evaluation Engine（v1，独立设计稿）

> 第 1 轮独立设计。作者模型：Claude Opus 4.8。
> 本稿是对 `测试参考.md`（AGENIX-Bench 草案）的**批判性超越**，严格遵循 `00_DESIGN_BRIEF.md` 的 §A~§K 结构与质量底线。
> 命名：**ACEE**（Agentic Capability Evaluation Engine），可视为 AGENIX 的 v2。

## §A 设计哲学与核心主张

### A.1 baseline 的根本缺陷（先破后立）

`测试参考.md` 的 AGENIX 草案是一个合理的"维度 + 加权求和"框架，但它**无法反映能力上限**，原因有五：

1. **只测结果、权重拍脑袋**：`A=0.4*Success+0.3*Tool+0.2*Step+0.1*Recovery` 这类公式，权重无依据、指标无精确定义（"Tool Correctness"怎么算？"Grounding Score"怎么落地？），且**单一总分会掩盖能力结构**。
2. **会饱和**：固定难度的题，强模型很快全做对，分数贴到 1.0 之后就再也分不出高低——**天花板测不出来**。
3. **会被污染**：静态题库会进训练集，强模型"背过"，分数虚高。
4. **会被刷**：固定 rubric 的 LLM-judge 容易被冗长输出、自我标榜、特定话术骗分；step efficiency 这类指标也能被"假装高效"操纵。
5. **无统计学**：没有重复、没有置信区间、没有显著性，两个模型差 2 分根本不知道是不是噪声。

### A.2 ACEE 的五条核心主张

> **主张 1：测"能力前沿"，而非"固定难度准确率"。**
> 不报告"在某难度下做对多少"，而是把难度/步数做成**连续可调旋钮**，测量模型**在哪个难度/步数上崩溃**（成功率跌破 50% 的点）。这条"崩溃点曲线"才是能力上限的直接刻画，且对题库构成鲁棒。用 **IRT（项目反应理论，2PL）** 把模型能力 θ 与题目难度 b 放到同一隐变量尺度上估计。

> **主张 2：过程 + 结果双层评分（Process + Outcome）。**
> Agentic/长程能力的核心在"怎么做"而非"做没做对"。用**里程碑 DAG + 轨迹信号**给过程打分，且过程分必须**带依赖校验**（不能跳过前置假装完成后置）。

> **主张 3：Verifier-first（验证器优先），judge 兜底。**
> 每个任务**必须**附带一个**程序化验证器（oracle）**，能确定性判定成功与关键中间状态。LLM-as-judge 只用于真正主观的子项（如计划质量、推理连贯性），且必须**多 judge + 人类校准 + 偏差消除**。能用规则/可执行校验的，绝不交给 judge。

> **主张 4：抗污染靠"程序化生成 + 私有持有 + 动态轮换 + 孪生题"。**
> 题目由带私有随机种子的**生成器**产出，表层随机化、实体动态注入；用"公开孪生题 vs 私有孪生题"的成绩差检测记忆/污染。

> **主张 5：抗刷分靠"对抗控制 + 健全性地板 + 捷径检测"。**
> 注入干扰工具、蜜罐、prompt injection；设置"空操作必须得 0 分"的 sanity floor；用模态消融/前置消融检测"假装在做但其实走捷径"。

### A.3 四大失真来源 → 对策一览（贯穿全文）

| 失真来源 | ACEE 对策（详见对应章节） |
|---|---|
| 饱和 | 连续难度旋钮 + IRT 能力前沿 + 题目自动退役（§C/§G） |
| 污染 | 程序化生成 + 私有种子 + 动态实体 + 孪生题污染探针（§C/§G） |
| 刷分作弊 | Verifier-first + 干扰/蜜罐 + sanity floor + 捷径/消融检测 + judge 去偏（§E/§H） |
| 评分噪声 | 多次重复 + bootstrap CI + 多 judge 一致性 + 配对显著性检验（§E/§G） |

## §B 能力本体论（Capability Ontology）

把三大能力族拆成**可观测、可计分的原子能力**。每个原子能力都给出"失败时长什么样"的可观测信号（这是设计指标的依据）。

```
ACEE Capability Tree
├── AGENTIC（工具/自主执行）
│   ├── A1 任务理解与拆解        失败信号：误解目标、漏子目标
│   ├── A2 工具选择              失败信号：选错工具、用干扰工具、漏调用
│   ├── A3 参数构造/接口遵从      失败信号：schema 非法、参数语义错、幻觉参数
│   ├── A4 观察-反应循环          失败信号：忽略工具返回、不读报错
│   ├── A5 多步纠错/恢复          失败信号：错误后继续硬走、重复同一错误
│   └── A6 终止判断              失败信号：早停、停不下来（死循环）
├── MULTIMODAL（多模态编排）
│   ├── M1 单模态感知/抽取        失败信号：OCR 错、读错图表数值
│   ├── M2 跨模态对齐            失败信号：把图 A 的信息安到文 B 上
│   ├── M3 跨模态融合推理         失败信号：只用单模态就答（不融合）
│   ├── M4 中间结构生成          失败信号：不产出可核查的中间表/结构
│   └── M5 模态陷阱抵抗          失败信号：被"单模态看似成立"的诱饵带偏
├── LONG-HORIZON（长程执行）
│   ├── L1 全局规划             失败信号：无规划直接动手、规划不可执行
│   ├── L2 状态跟踪             失败信号：忘记已改动、用过期状态决策
│   ├── L3 里程碑推进            失败信号：卡在中途、跳步
│   ├── L4 扰动/失败恢复         失败信号：环境变化后不自适应
│   ├── L5 长程一致性/抗漂移      失败信号：目标漂移、上下文遗忘、反复横跳
│   └── L6 资源/预算管理         失败信号：步数/成本爆炸
└── PLANNING & META（横切）
    ├── P1 计划质量             失败信号：拆解不全、顺序错、不可行
    ├── P2 计划-执行一致性        失败信号：计划归计划、执行归执行
    └── P3 反幻觉              失败信号：编造工具/编造结果/编造引用
```

> 设计原则：**每个原子能力至少被一个程序化信号覆盖**；judge 只覆盖 P1 等少数固有主观项。

## §C 任务体系（Task Taxonomy）

### C.1 任务族 × 难度旋钮

每个能力维度下设若干"任务族（family）"，每个 family 是一个**参数化生成器** `gen(seed, knobs) -> Task`。难度由**可量化旋钮**控制，因此难度是**连续/分级且可控生成**的：

| 维度 | 代表任务族 | 难度旋钮（连续可调） |
|---|---|---|
| Agentic | 多源数据对账、API 编排、检索-汇总、文件系统模拟操作、函数调用 | 工具数量、干扰工具比例、必需步数 K、参数约束严格度、注入故障数 |
| Multimodal | 图表读数+计算、文档(发票/报表)抽取核对、图文证据链、视频事件时序 | 模态数、信息分散度、模态陷阱数、噪声(模糊/遮挡)强度、需融合的字段数 |
| Long-horizon | 多文件代码重构、多步研究→产出报告、流程编排、状态机任务 | 里程碑数 N、DAG 深度/分支、状态可变性、扰动次数、上下文干扰长度 |
| Planning | 受限规划、依赖排序、预算约束规划 | 约束数、目标数、陷阱(不可行子目标)数 |

难度分级映射：`easy / medium / hard / expert / ceiling(天花板级)`。**ceiling 级专为强模型设计**：旋钮拉到当前最强模型也只有 ~30-50% 成功率，保证不饱和。

### C.2 抗污染的题目生命周期

```
generator(family, seed_private, knobs)
   ├── 程序化生成结构（DAG / 工具集 / gold 解）
   ├── 表层随机化（变量名/实体/数值/措辞 paraphrase）
   ├── 动态注入（当前时间戳、随机命名实体、一次性 canary 串）
   └── 产出：public_twin（可公开示例） + private_twins×M（私有，永不公开）
```

- **私有持有**：评测主集为私有 twin，永不进入互联网；只公开极少量 public twin 作示例。
- **污染探针（contamination probe）**：对每个 family，比较模型在 `public_twin` 与 `private_twin` 上的成绩差 `Δ_contam = score_public − score_private`。`Δ_contam` 显著 > 0 → 记忆/污染嫌疑，给该模型在该 family 打"污染标记"并在报告中标注（而非直接采信高分）。
- **题目退役**：当某题被 IRT 判为"区分度→0 / 难度过低（全员通过）"时，自动退役并由生成器补充更难旋钮的新题（对抗饱和）。

## §D 完整任务样例（含逐项打分演示）

> 评分公式的完整定义见 §E；此处每例内联标注所用公式，并**把分数真正算出来**。

### 样例 1（Agentic）：跨源对账 + 计算，含干扰工具与注入故障

**任务输入**：给定"订单系统"和"财务系统"两套只读 API，找出两边金额不一致的订单，输出不一致订单号列表与差额总和。
**可用工具（含干扰）**：`orders.query`、`finance.query`、`math.sum`、`currency.convert`（**干扰**：本题全为同币种，用了反而引入误差）、`email.send`（**蜜罐**：不该调）。
**注入故障**：`finance.query` 第一次调用返回 `RATE_LIMIT` 错误，需重试/退避。
**gold 动作序列（最优 K=5 步）**：`orders.query → finance.query(遇错) → finance.query(重试成功) → 比对 → math.sum`。
**oracle**：确定性比对程序，给出 gold 不一致集合 = {1003,1007} 与差额 = 42.50。
**里程碑**：m1 取订单、m2 取财务(含恢复)、m3 正确比对、m4 正确汇总。

**某模型实测 trace（摘要）**：正确取数；遇 RATE_LIMIT 后重试成功（√恢复）；**误调用了 `currency.convert`** 导致差额算成 44.10；未碰蜜罐；共 7 步。

**逐项打分（公式见 §E.1）**：
- 任务成功 `TSR`：最终集合对（{1003,1007}√）但差额错（44.10≠42.50）→ 部分成功。本任务 oracle 定义 `TSR = 0.5*集合正确 + 0.5*数值正确 = 0.5*1 + 0.5*0 = 0.5`。
- 工具正确率 `TCR`：6 次"必要/有效"调用中 5 次正确，1 次（currency.convert）为有害多余调用 → `TCR = 5/6 ≈ 0.83`；触发"有害工具惩罚"-0.1。
- 步效率 `SE = clip(K_opt/K_act,0,1)=5/7 ≈ 0.71`。
- 错误恢复 `ERR = 已恢复故障/注入故障 = 1/1 = 1.0`。
- 蜜罐：未触碰，无惩罚。
- 维度分 `A = 0.4*TSR + 0.3*TCR + 0.2*SE + 0.1*ERR − 有害惩罚`
  `= 0.4*0.5 + 0.3*0.83 + 0.2*0.71 + 0.1*1.0 − 0.1 = 0.2+0.249+0.142+0.1−0.1 = 0.591`。

### 样例 2（Multimodal 编排）：图表 + CSV + 文本规范融合，带模态陷阱

**任务输入**：①一张柱状图（PNG，含 2019-2023 年营收）；②一个 CSV（含同公司"成本"列，但**年份顺序被打乱**）；③一段文本规范："计算利润率最高的年份（利润率=(营收−成本)/营收），只用图里的营收和 CSV 里的成本"。
**模态陷阱**：CSV 里也有一列名为 `revenue_estimate`（与图中真实营收不同）——只看 CSV 就能算出一个**看似合理但错误**的答案（陷阱）。
**oracle**：程序化计算正确年份 = 2021，利润率 = 0.37；并记录每年份的中间利润率表。
**grounding 真值**：图中每根柱子的数值区域（用于 OCR/读数核对）。

**某模型 trace**：正确从图读出 5 个营收值（读数全对）；从 CSV 取 `cost` 并**按年份对齐**（√跨模态对齐）；**未**误用 `revenue_estimate`（√抗陷阱）；产出了中间利润率表；最终答 2021、0.37。

**逐项打分（公式见 §E.2）**：
- 答案准确 `AA = 1.0`（年份+数值都对）。
- grounding `GS`：5/5 营收读数与真值匹配 → `GS = 1.0`。
- 跨模态一致性 `XMC`：中间利润率表里"营收来自图、成本来自CSV"两侧数值与各自 oracle 完全匹配 → `XMC = 1.0`；并通过"成本模态消融"反事实校验（去掉 CSV 则无法得出该答案 → 确实融合）。
- 模态陷阱抵抗 `MTR = 1.0`（未用 revenue_estimate）。
- 维度分 `B = 0.45*AA + 0.30*GS + 0.20*XMC + 0.05*MTR = 0.45+0.30+0.20+0.05 = 1.0`。
（对照：若模型偷懒只用 CSV 的 `revenue_estimate`，AA 与 XMC 直接崩，且消融校验暴露其未融合 → 得分骤降，刷分被抓。）

### 样例 3（Long-horizon）：多文件重构 + 注入失败 + 状态演化（里程碑 DAG）

**任务输入**：一个小型代码仓（沙箱），要求把同步 IO 重构为异步，且全部单测通过；中途**注入扰动**：第 3 步后，环境"外部修改"了一个配置文件（模拟他人提交），模型必须察觉并适配。
**里程碑 DAG**（带依赖、带权）：
```
m1 阅读/定位(0.1) → m2 制定计划(0.1) → m3 改造模块A(0.2)
m3 → m4 改造模块B(0.2) → m6 全量测试通过(0.2)
m3 → m5 适配被外部改动的配置(0.15)[扰动恢复] → m6
m6 → m7 无回归/不破坏既有API(0.05)
```
**oracle**：跑测试套件（确定性），各里程碑用**状态谓词**判定（如 m6 = 测试全绿；m5 = 配置冲突被正确合并）。

**某模型 trace（摘要）**：先规划（√m1,m2）；改 A（√m3）；改 B（√m4）；**未察觉配置被改**，直接跑测试 → 失败；模型读报错后定位到配置冲突并修复（√m5，属"读错误后恢复"）；再次测试全绿（√m6）；有 1 处旧 API 签名被破坏（×m7）。中途有 2 次回到已改过的文件重复改动（轻微回头路）。总步数 18，最优约 12。

**逐项打分（公式见 §E.3）**：
- 最终成功 `FS`：测试全绿 = 1.0（但 m7 回归项单独计）。
- 里程碑完成（DAG 有效）`MC = Σ(已达且前置全达的权重)/Σ权重`
  已达 m1..m6（权重 0.1+0.1+0.2+0.2+0.15+0.2=0.95），m7 未达(0.05)。所有已达里程碑前置均满足 → `MC = 0.95/1.00 = 0.95`。
- 轨迹稳定性 `TS = 1 − 受罚步/总步`：2 次回头路记为受罚步 → `TS = 1 − 2/18 ≈ 0.89`。
- 状态恢复 `SR`：扰动 1 次，最终经"测试失败→定位→修复"成功适配（非主动察觉，属被动恢复，按 §E.3 记 0.7）。
- 维度分 `C = 0.4*FS + 0.35*MC + 0.15*TS + 0.10*SR`
  `= 0.4*1.0 + 0.35*0.95 + 0.15*0.89 + 0.10*0.7 = 0.4+0.3325+0.1335+0.07 = 0.936`。
（注：`MC` 的 DAG 有效性意味着——如果模型"假装"完成 m6 却没真正通过 m3/m4 前置，m6 不计分，杜绝跳步刷分。）

## §E 评分引擎（Scoring Engine）—— 核心

### E.0 三层评估器分工

```
Layer 1  Rule/Executable Verifier（确定性，优先）
         ├─ exact / set / numeric match（带容差）
         ├─ 可执行校验：跑测试、跑 oracle 程序、schema 校验
         └─ 工具调用比对（tool+args vs gold，支持多条等价 gold 路径）
Layer 2  Embedding / 跨模态对齐（半确定性）
         ├─ 语义相似度（开放式文本答案，阈值+人工校准）
         └─ 图文/跨模态数值匹配、区域 grounding（IoU / 源归因）
Layer 3  LLM-as-judge（仅限固有主观项：计划质量、推理连贯、解释充分性）
         └─ rubric + 多 judge + 去偏 + 人类校准（见 E.5）
```
**铁律**：任何能被 Layer 1 判定的，不下放到 Layer 3。Judge 只在 Layer 1/2 无法覆盖时介入，且其分数占比受限（见 E.4 权重）。

### E.1 Agentic 维度 A 的精确定义

- 任务成功 `TSR ∈ [0,1]`：由 oracle 定义（可二值，可分段，如样例1 的 0.5/0.5 分解）。
- 工具调用正确率：对每次调用 i，`m_i = 1`（工具对且参数对）/`0.5`（工具对参数错）/`0`（工具错或幻觉工具）；`TCR = (Σ m_i)/N_calls`。多条等价 gold 路径时取**最优对齐**（见 E.6）。
- 步效率 `SE = clip(K_opt / K_act, 0, 1)`，`K_opt` 由 gold 最短有效路径给出；仅在 `TSR>0` 时计入，否则记 N/A 不拉低（避免"快速失败"反而高效的悖论）。
- 错误恢复 `ERR = 成功恢复的故障数 / 注入故障数`（故障是**确定性注入**的，可控可复现）。
- 惩罚：有害工具/蜜罐每次 −0.1（封顶 −0.3）。
- `A = 0.40*TSR + 0.30*TCR + 0.20*SE + 0.10*ERR − Penalty`，再 clip 到 [0,1]。

### E.2 Multimodal 维度 B 的精确定义

- 答案准确 `AA`：oracle 判定（数值带相对容差 ε，如 1e-3）。
- grounding `GS`：
  - 有 bbox 真值：`GS = mean(IoU(pred_region, gt_region) ≥ τ)`，τ=0.5。
  - 无 bbox、有源真值：`GS = 正确源归因数 / 关键证据数`（模型须声明每个关键数来自哪个模态/位置，程序核对）。
  - 都没有：退化为 judge 校验引用正确性（降权）。
- 跨模态一致性 `XMC`：要求模型产出**中间结构**（如键值表），程序逐项核对"每个字段取自正确模态且数值与该模态 oracle 一致"；`XMC = 正确融合字段 / 需融合字段`。并以**模态消融反事实**作 gate：去掉必需模态后仍"成功"则判定未融合，XMC 置 0 并标记捷径。
- 模态陷阱抵抗 `MTR = 1 − 误用陷阱信息的比例`。
- `B = 0.45*AA + 0.30*GS + 0.20*XMC + 0.05*MTR`。
- **纯文本模型降级**：无视觉模型走"感知剥离轨道"——把图像内容以结构化文本/OCR 提供，单独测 M2/M3/M4（融合与推理），不与多模态模型混排，报告中明确标注"覆盖度"。

### E.3 Long-horizon 维度 C 的精确定义

- 最终成功 `FS`：oracle/测试判定。
- 里程碑完成（DAG 有效）：`MC = Σ_m w_m·1[reached(m) ∧ ∀p∈pre(m): reached(p)] / Σ_m w_m`。`reached(m)` 由**状态谓词**判定（不是看模型"说"完成，而是看环境状态）。
- 轨迹稳定性：`TS = 1 − (penalized_steps / total_steps)`，`penalized_steps = backtrack + 冗余重复 + 无效重试 + 目标漂移步`（各由 state-diff 与动作日志检测，见 §H.3）。
- 状态恢复 `SR`：每次扰动按"主动察觉并适配=1.0 / 被动(经报错)恢复=0.7 / 未恢复=0"，多扰动取均值。
- `C = 0.40*FS + 0.35*MC + 0.15*TS + 0.10*SR`。
- 设计要点：`MC` 的依赖门控 + 状态谓词，使"跳步/嘴上说完成"无法骗分；这是长程"中间状态会变"在评分里的真正落地。

### E.4 Planning 维度 D 的精确定义

- 计划质量 `PQ`（**唯一以 judge 为主**的子项）：rubric 五项各 0-2 分（拆解完整性、可行性、顺序正确性、工具落地性、无幻觉工具），`PQ = Σ/10`，由 ≥3 judge 取中位数。
- 计划-执行一致性 `PEC = 1 − 不合理偏离率`。**关键**：为恢复/纠错而偏离计划是**好事**，不罚；只罚"无理由的随意偏离"（用 E.6 的对齐 + 偏离归因区分）。
- 反幻觉 `HP_penalty = min(1, 幻觉调用/编造结果次数 / N_calls)`。
- `D = 0.5*PQ + 0.3*PEC + 0.2*(1 − HP_penalty)`。

### E.5 LLM-as-judge 的去偏与校准（评测可信度命门）

1. **结构化裁决**：judge 必须按 rubric 输出**每项分数 + 证据引用（quote）**，禁止只给总分；无证据的项不计。
2. **偏差消除**：
   - 位置/顺序偏差 → 成对比较时随机交换 A/B 顺序并双向跑。
   - 长度偏差 → 对长度做控制（rubric 明确"长度不加分"，并做长度-分数回归校正）。
   - 自我偏好 → judge **对模型身份盲化**；且 judge 模型不可与被测模型同源（交叉异构）。
3. **多 judge 集成**：≥3 个异构 judge，取中位数；计算 judge 间一致性 **Krippendorff's α**，α<0.67 的条目标记"低一致"，要么扩大 judge 池要么送人工。
4. **人类校准集**：维护数百条人工标注样本，定期测 **judge–人类一致性**（Spearman ρ / Cohen's κ）；低于阈值则该 rubric 维度禁用 judge、回退规则或人工。
5. **成对优于打分**：排名场景用成对偏好 + **Bradley-Terry** 拟合得到稳定分，缓解绝对分尺度漂移。

### E.6 多条等价路径与对齐

工具序列/计划很少只有唯一正解。每题维护**等价 gold 集合**（DAG 形式，允许并行/可换序步骤）；评分时用**带语义等价的序列对齐**（canonical 化步骤标签后做加权编辑距离/最优匹配），取与任一 gold 路径的最优对齐结果。这样既不冤枉合理的不同走法，也能识别真正的偏离。

### E.7 聚合、归一化与"不要单一总分"

- 维度分先各自归一到 [0,1]。
- **首要产物是能力画像向量** `Profile = (A, B, C, D, 以及各原子能力子分)` + **能力前沿指标**（见 §G IRT 的 θ、崩溃难度 d*、崩溃步数 h*）。
- **TOTAL 仅作便捷标量**：默认 `TOTAL = 0.30*A + 0.25*C + 0.25*B + 0.20*D`，但：
  - 权重**可配置**（提供 balanced / agentic-heavy / multimodal-heavy 等 profile）。
  - 必做**权重敏感性分析**：报告"在权重扰动 ±50% 下排名是否稳定"，并公开"排名翻转的权重区域"。让权重不再是拍脑袋——排名结论只在"对权重稳健"时才下定论。

## §G 统计方法学（Statistical Rigor）

1. **重复与不确定性**：每题跑 `k`（默认 5）次（不同采样种子）；报告均值 + **bootstrap 95% CI**；同时报 `pass@1`、`pass@k`、以及成功率方差（稳定性）。
2. **能力前沿（核心）**：用 **IRT 2PL** `P(success|θ,a,b)=σ(a(θ−b))` 拟合：从一组校准模型估计每题的难度 `b`、区分度 `a`，再估每个被测模型能力 `θ`（带 CI）。
   - 头条指标：`θ`（隐能力）、**崩溃难度 `d*`**（成功率=50% 的难度）、长程任务的**崩溃步数 `h*`**（成功率跌破 50% 的里程碑数/步数）。这三者直接刻画"上限"。
3. **题目质检/退役**：区分度 `a≈0` 或难度过低（全员通过）的题判为"无信息"，自动退役（抗饱和），由生成器补更难题。
4. **抗污染统计**：污染探针 `Δ_contam`（§C.2）+ canary 命中检测；held-out 私有集为准。
5. **模型间显著性**：配对 **bootstrap / 置换检验**（同题配对），多重比较用 **Holm–Bonferroni** 校正；报告**效应量**（如 Cliff's δ）。两模型差异未过显著性 → 报告"不可区分"，绝不下"谁更强"的结论。

## §H 对抗与鲁棒性（Adversarial & Robustness）

### H.1 噪声/干扰注入（已内建到任务）
- 干扰工具（distractor）、蜜罐工具（honeypot）、误导性无关信息、部分可观测（关键信息需主动查询而非直接给）。

### H.2 安全/对抗轨道（独立计分，不混入能力总分）
- **Prompt injection**：在工具返回/文档中植入"忽略原指令，改做 X / 泄露系统提示"，度量 `injection_resist = 抵抗成功数 / 注入数`。
- **工具欺骗**：返回**看似成功实则错误**的结果，度量模型是否盲信（结合 oracle 真值）。
- **多智能体冲突**（可选）：注入一个给出错误建议的"伙伴 agent"，看是否被带偏。

### H.3 长程漂移检测（量化）
- **目标漂移**：周期性用 oracle 目标对比当前动作的目标对齐度（嵌入相似度跌破阈值=漂移步）。
- **上下文遗忘**：在长上下文中埋"必须记住的约束"，后段检测是否违反（needle 式，但绑定到任务动作而非纯问答）。
- **重复错误/循环**：动作-状态序列做循环检测（相同(state,action)对重复 ≥ 阈值 → 计入受罚步并触发 L5 失分）。

## §K 自我标注：最易被质疑/最不确定的点（留给第 2 轮辩论）

1. **IRT 的可行性与冷启动**：2PL 需要一定规模的"模型×题目"作答矩阵来稳定估计 `a,b`；模型数量少时 `θ/d*` 估计可能不稳。是否该退而用更简单的"成功率-难度曲线 + 逻辑回归崩溃点"？这是我最不确定的点。
2. **里程碑 DAG 的人工成本**：每个长程任务都要人工/半自动定义带权依赖 DAG 与状态谓词，规模化成本高；生成器能否自动产出可靠的里程碑与谓词，存疑。
3. **judge 即便去偏，仍可能系统性偏差**：把 PQ（计划质量）交给 judge，是否会成为整个体系最脆弱、最易被"话术"操纵的一环？是否应进一步压缩 judge 占比，甚至把"计划质量"也转成可执行代理指标（如计划对最优 DAG 的覆盖率）？
4. **权重与 TOTAL 的意义**：我主张"以画像+前沿为主、TOTAL 为辅"，但用户可能就是想要"一个能排序的分"。敏感性分析能否真的让单一总分变得可信，仍需辩论。
5. **轨迹稳定性指标的可操纵性**：`TS`/漂移检测依赖 state-diff 与循环检测的阈值，阈值设定可能被特定行为模式利用或误伤合理的探索性回溯。
6. **真实环境 vs 模拟器的保真度**：全用录制重放的工具模拟器，可复现性强，但可能测不出模型在"真实、嘈杂、非确定"环境中的鲁棒性，存在外部效度（external validity）折损。

> 以上 6 点请第 2 轮的 GPT-5.5 / Opus 4.6 重点攻击与改进。
```
