# 共享设计纲要（DESIGN BRIEF）—— 多模型协同设计「Agentic Test Engine」

> 本文件是多模型协同设计的统一纲要。GPT-5.5 / Opus 4.6 / Opus 4.8 三个模型在第 1 轮各自**独立**按本纲要产出一份完整设计，第 2 轮互相质疑，第 3 轮融合定稿。请严格遵守本纲要的章节结构、质量底线与输出约定。

---

## 0. 总目标（用户的真实诉求）

设计一个能够**深度测试大模型能力上限**的统一评测体系 / **test engine**，覆盖三大能力族：

1. **Agentic 能力**：任务拆解、工具调用、多步纠错、自主决策。
2. **多模态任务编排能力**：图 / 文 / 表 / 文档 / 视频跨模态串联推理与编排，中间产物生成，grounding。
3. **长程任务能力（long-horizon）**：10~50+ 步、中间状态会变、需要失败恢复、需要保持长程一致性与目标对齐。

**硬性要求：**

- 必须**自动化**：端到端可跑，无需人工逐题判分。
- 必须**可量化**：每个能力维度都有精确、可复现的分数。
- 必须能**横向对比多个模型**（被测对象包括 seed / deepseek / kimi / glm / GPT / Claude 等，通过 API 访问）。
- 必须能**真正反映能力上限**：抗饱和、抗污染、抗刷分、抗评分噪声。

用户特别强调：**本设计没有上下文长度与思考深度的上限**，要尽可能详细、深入、完善。宁可冗长也不要含糊。

---

## 1. 关键约束与现实条件（设计时必须考虑）

- **被测模型能力参差不齐**：有的无视觉（纯文本）、有的无原生 function-calling、上下文窗口差异大、有的不支持长输出。设计必须说明：当某能力缺失时如何降级测试 / 如何公平计分（例如纯文本模型在多模态维度如何处理）。
- **工具与环境**：真实环境（真浏览器 / 真 OS）成本高、不稳定、难复现。需要在「真实环境保真度」与「可复现性 / 可自动化」之间做明确权衡，建议以**可复现的沙箱 / 工具模拟器 / 录制重放**为主，必要时给出真实环境的可选扩展。
- **评测可信度是第一性问题**：如果评分不稳定、可被刷、被污染，那么"反映能力上限"就是空话。请把这一点当作设计的核心而非附加项。

## 2. 必须批判性超越的 baseline

工作区文件 `d:\文档\Cursor\model test\测试参考.md` 是一份草案（AGENIX-Bench：维度 A 工具使用 / B 多模态编排 / C 长程 / D 规划，加权求和 TOTAL）。

请先读它，把它当作 **baseline 草案**。你的任务是**批判性地超越**它，而不是照抄。请至少明确指出 baseline 的若干不足，例如（不限于）：

- 权重（0.4/0.3/0.2/0.1 等）凭什么这么定？是否有依据？是否会被刷？
- "Tool Correctness""Step Efficiency""Grounding Score"等指标的**精确定义和可计算实现**是否缺失？
- LLM-as-judge 如何防偏差、如何自校验、judge 与人类一致性如何验证？
- 如何防数据污染 / 防过拟合 / 防刷分？baseline 几乎没谈。
- 统计显著性、方差、重复实验、置信区间是否缺失？
- 长程"中间状态会变"如何在评分里真正体现？

## 3. 统一输出模板（每个模型必须按此结构产出）

请用中文，分以下章节，**充分展开**：

**§A 设计哲学与核心主张**
- 这套 test engine 为什么能反映"能力上限"？核心机制是什么？
- 如何系统性对抗：benchmark 饱和、数据污染、刷分作弊、评分噪声。

**§B 能力本体论（Capability Ontology）**
- 把 Agentic / 多模态编排 / 长程 拆成可测的**原子能力维度**（建议给出层级化的能力树），每个原子能力配上"它失败时长什么样"的可观测信号。

**§C 任务体系（Task Taxonomy）**
- 每个能力维度下的任务族、难度分级（建议含 easy/medium/hard/expert/**ceiling 天花板级**），难度是如何"可控生成 / 可参数化"的。
- 说明任务如何做到**抗污染**（程序化生成 / 私有持有 / 动态参数化 / 定期轮换）。

**§D 完整任务样例（至少 3 个，跨维度）**
- 每个样例必须给出：任务输入、可用工具集（含干扰工具）、gold 解 / gold 动作序列、milestone DAG、期望 trace、以及**逐项打分演示**（把分数真的算出来）。
- 至少 1 个长程样例（体现中间状态变化 + 失败恢复），至少 1 个多模态编排样例（体现跨模态串联 + grounding）。

**§E 评分引擎（Scoring Engine）——本设计的核心**
- 每个指标的**精确数学定义**（给公式）、聚合方式、权重及其**依据**。
- 三层评估器如何分工：(1) rule-based / 可执行校验；(2) embedding / 语义与跨模态对齐；(3) LLM-as-judge。
- LLM-as-judge：给出 rubric 模板、防偏差措施（位置偏差、长度偏差、自我偏好等）、**多 judge 一致性**与**judge–人类一致性**的量化校验方法（如 Cohen's κ / 相关系数），以及 judge 失效时的兜底。
- 给出"分数如何归一化、如何聚合成能力画像"的方案，并论证权重不是拍脑袋。

**§F 运行时与数据契约（Runtime & Schemas）**
- harness 总体架构（被测模型适配层 / 工具与环境层 / trace 记录层 / 评估层 / 报告层）。
- 统一 **Task Schema**（JSON）、统一 **Trace Schema**（记录 tool calls、中间产物、推理步骤、状态快照、最终输出）。
- 沙箱 / 工具模拟 / 可复现性（随机种子、版本钉死、录制重放）。
- 如何适配"无视觉 / 无原生 tool-calling / 不同上下文窗口"的模型。

**§G 统计方法学（Statistical Rigor）**
- 重复次数、pass@k、置信区间、方差报告、能力曲线（随难度/步数的衰减曲线）。
- 防过拟合 / 防污染的统计手段（如 held-out、canary、动态题库）。
- 模型间差异的**显著性检验**。

**§H 对抗与鲁棒性（Adversarial & Robustness）**
- 噪声注入（无关工具、误导信息、部分可观测）。
- prompt injection / 工具欺骗 / 多智能体冲突。
- 长程"漂移检测"（目标漂移、上下文遗忘、重复错误）的量化指标。

**§I 多模型横向对比与报告**
- 标准化 / 归一化、雷达图 / 能力画像、排行榜设计、显著性标注。
- 如何避免"单一总分掩盖能力结构差异"。

**§J 工程落地路线**
- 目录结构、关键模块接口（伪代码即可）、最小可跑 MVP 的里程碑与优先级。

**§K 自我标注：最易被质疑/最不确定的点**
- 明确列出你这份设计中 3~5 个**最可能被其他模型挑刺**或你自己最不确定的设计选择，方便第 2 轮辩论。

## 4. 质量底线（不满足视为不合格）

1. 评分必须"**可自动计算 + 可复现**"，每个指标都要能落到代码层面（给出计算逻辑/伪代码）。
2. 必须正面、深入地解决"**benchmark 饱和 / 数据污染 / 刷分 / 评分噪声**"四大失真来源。
3. 至少 3 个**完整且把分数算出来**的任务样例。
4. 长程维度必须体现"中间状态会变 + 失败恢复 + 里程碑 DAG + 轨迹稳定性"。
5. 多模态维度必须体现"跨模态串联 + grounding 量化"。
6. 必须给出 harness 架构 + Task/Trace schema + 目录结构 + 关键伪代码 + MVP 路线。
7. 全程**中文**，结构清晰，宁深勿浅。

## 5. 协作流程说明（你在第几轮）

- **第 1 轮（当前）**：你**独立**设计，**不要**读取其它模型的设计文件（`01_design_*.md`）。把可被挑刺的细节摆出来。
- **第 2 轮**：你会拿到另外两个模型的设计，进行批判性 critique（找漏洞、找可刷分点、找不可复现处、找覆盖盲区），并指出三份设计中每个维度的"最优解"。
- **第 3 轮**：由一个模型融合所有设计与质疑，产出最终统一规范。

请在第 1 轮把设计做到"足以被认真挑战"的深度。
